# AGENTS.md — Repository context for AI coding agents

## What this repo is

A **runnable demo** that proves an MCP client can authenticate automatically to a secured MCP
gateway using the [MCP authorization spec](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)
with Red Hat SSO as the identity provider.

The infrastructure (Envoy / Istio + Kuadrant AuthPolicy) is provided by
[MCP Gateway](https://github.com/Kuadrant/mcp-gateway). Token issuance is handled by the hosted
[MCP Auth Adapter](https://github.com/velias/mcp-auth-adapter), which implements DCR and
Authorization Code + PKCE in front of Red Hat SSO.

**Target audience:** external / open-source community — anyone building or evaluating MCP gateway
deployments with OAuth.

## What this repo is NOT

- Not a production-ready deployment — it is a proof of concept.
- Not Red Hat Insights tooling — `insights-mcp` is a **stand-in backend**. The gateway and auth
  infrastructure are completely backend-agnostic. Any MCP server can be registered.
- Not the source code of MCP Gateway or MCP Auth Adapter — those live in their own repos
  (linked above). This repo only contains deployment scripts, manifests, and configuration.
- Not a library or SDK — nothing here is imported by other projects.

## Two questions this demo answers

1. **Can an MCP client authenticate automatically** via DCR + PKCE against a secured gateway?
   → Yes. The `kind/` deployment proves this with Claude Code (`http://localhost:8001/mcp`).
   Cursor has a known localhost OAuth bug — a Bearer workaround exists.

2. **Can the resulting token call the Red Hat Insights API?**
   → Yes, with either `env.stage` or `env.prod` and scopes `api.console api.ocm`.
   The account must have the [RBAC roles required by each toolset](https://github.com/RedHatInsights/insights-mcp/tree/main#required-permissions-by-toolset)
   assigned in its organization.

## Key architectural facts

- **MCP Auth Adapter is always external** — it never runs locally. Both deployments point at the
  hosted instance via `MCP_AUTH_BASE`.
- **The gateway does not proxy OAuth** — it only publishes Protected Resource Metadata (PRM) at
  `/.well-known/oauth-protected-resource`. Clients call the adapter directly.
- **Two credential roles exist** — the broker token (used by the broker for `tools/list` with
  upstream servers) and the client token (used by MCP clients for `tools/call`). They are distinct.
- **Auth enforcement differs by deployment:**
  - `local-deployment`: pass-through — each upstream server validates the JWT.
  - `kind`: Kuadrant AuthPolicy enforces JWT at the Istio gateway before traffic reaches the broker.
- **insights-mcp is replaceable** — registering a different backend requires only a credential
  Secret, an HTTPRoute, and an `MCPServerRegistration` (kind) or a `config.yaml` entry (local).

## Environment selection

Two env files at the repo root control which Red Hat SSO instance is used. Always source one
before running any script:

```bash
source env.prod    # production SSO + Insights API
source env.stage   # stage SSO + Insights API
```

## Scope for contributions / modifications

In scope:
- Improvements to deployment scripts (`setup.sh`, `start.sh`, `refresh-token.sh`, etc.)
- Additional manifests or configuration for registering new MCP backend servers
- Documentation improvements (README files, inline script comments)
- Troubleshooting additions based on newly encountered failure modes

Out of scope:
- Changes to MCP Gateway source code (lives in `Kuadrant/mcp-gateway`)
- Changes to MCP Auth Adapter source code (lives in `velias/mcp-auth-adapter`)
- Insights-specific business logic — `insights-mcp` is referenced only as an example
- New Python scripts beyond `get-token.py` and `inspect-token.py` (avoid scope creep)

## Conventions

- Shell scripts use `bash` and source env files from the repo root before running.
- Python scripts require Python 3; no virtual environment is assumed — keep dependencies minimal.
- `config.runtime.yaml` is gitignored — it is generated by `start.sh` with the real token
  substituted in; never commit it.
- Pinned versions for all Helm charts and Kubernetes components are documented in the root
  `README.md#pinned-versions` section. Update those entries when bumping versions.
- Do not write Insights-specific logic into gateway manifests or scripts — keep them generic.
